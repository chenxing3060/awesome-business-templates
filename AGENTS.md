# 代理操作手册

你正在操作 **awesome-business-templates** 模板库。你的任务是根据用户的需求，从 205 套模板中选出最适合的风格，构建出漂亮的 HTML 演示文稿。

本文档是你的操作手册。在处理任何模板任务时，先通读一遍。

---

## 1. 完整工作流

对于每个演示文稿构建请求，请严格按照以下顺序执行。**不要跳过第 1 步（询问）和第 4 步（预览）**。

### 第 1 步 — 询问用户场景和情绪

在读取任何文件之前，先问用户两件事——即使需求看起来很明显：

> "在使用模板之前，我想了解两点：
> 1. **使用场景是什么？**（例如：作品集展示、教学演示、品牌宣传、画廊展览等）
> 2. **希望传达什么氛围？**（例如：温暖文艺、大胆前卫、静谧典雅、奢华精致等）"

等待用户回答。不要提前选择。用户的回答往往包含比初始需求更丰富的信息。

### 第 2 步 — 读取 `templates.json` 并挑选 3 个候选

读取 `templates.json`。用用户的场景和情绪对照每个模板的 `tags`。

**选出 3 个在情绪和风格上有明显差异的候选**。不要选三个同类的——例如，如果用户要"宁静"，不要选三个冷色调模板，而是选一个宁静的、一个温暖的、一个中性的作为对比。

`templates.json` 中每个模板的结构：

```json
{
  "id": "imp-001",
  "name": "光与色的诗篇",
  "baseTemplate": "impressionism",
  "description": "柔光渐变、可见笔触、暖色调",
  "tags": {
    "colorScheme": "warm",
    "mood": "serene",
    "movement": "impressionism",
    "formality": "medium",
    "density": "medium",
    "useCase": "gallery",
    "fontStyle": "display",
    "era": "vintage"
  },
  "variants": {
    "accentColor": "#e8a0b4",
    "bgTint": "#fef9f0"
  }
}
```

关键匹配维度（按优先顺序）：
- `mood`（情绪）—— 最匹配用户的感觉
- `movement`（流派）—— 用户如果提到特定风格
- `useCase`（场景）—— 用户要做什么用
- `colorScheme`（色彩）—— 浅/深/暖/冷等
- `formality`（正式度）—— 高/中/低
- `density`（密度）—— 内容量

### 第 3 步 — 为每个候选构建首屏预览

对于每个候选：

1. 读取基座模板 `base/<baseTemplate>.html`，了解其视觉系统（字体、颜色、布局、装饰）
2. 提取**第一页**（封面/标题页）
3. 将占位内容替换为用户的**真实内容**（标题、副标题、作者、日期）
4. 保存为独立 HTML 到 `templates/` 目录，例如 `templates/01-candidate-name.html`

每个预览文件应自包含——打开即可看到该模板的首屏。

### 第 4 步 — 在浏览器中打开所有 3 个预览，发送路径给用户

用 `open <path>` 打开每个文件。然后发消息：

> "三个候选风格，欢迎对比：
> 1. **<Candidate A>** — <一句话风格描述>
>    `/path/to/previews/01-a.html`
> 2. **<Candidate B>** — <一句话风格描述>
>    `/path/to/previews/02-b.html`
> 3. **<Candidate C>** — <一句话风格描述>
>    `/path/to/previews/03-c.html`
>
> 哪个感觉更对？"

等待用户选择。

### 第 5 步 — 在选定模板中构建完整演示文稿

一旦用户选定：

1. 复制基座模板到用户的 workspace
2. 按适配规则替换每一页的内容
3. 如果用户的内容需要更多页，复制现有布局扩展；如果少，从末尾删除
4. **更新所有页码标签**
5. **如果需要模板中没有的布局，用模板的设计系统从零设计**（见 §3）

### 第 6 步 — 打开最终演示文稿，发送文件路径

```bash
open /path/to/final/deck.html
```

然后发送消息，包含：

> "完成！你的演示文稿在 `/path/to/final/deck.html`——已打开。
>
> [一句说明用了哪个模板及注意事项]"

## 2. 如何读取 `templates.json`

`templates.json` 包含所有 205 套模板的元数据。结构如下：

| 字段 | 用途 |
|------|------|
| `id` | 唯一标识符 |
| `name` | 中文名称 |
| `baseTemplate` | 对应的基座模板文件名（不含 .html） |
| `description` | 一句话描述 |
| `tags.mood` | 情绪氛围。匹配用户的感觉关键词 |
| `tags.movement` | 艺术流派 |
| `tags.colorScheme` | 色彩方案 |
| `tags.useCase` | 推荐使用场景 |
| `tags.formality` | 正式程度 |
| `tags.density` | 内容密度 |
| `tags.fontStyle` | 字体风格 |
| `tags.era` | 时代感 |

## 3. 如何适配选定模板

### 始终保留（设计系统不可改）

- **字体** — 模板导入的 Google Fonts、font-family 声明。绝不替换
- **配色** — 所有 `:root` CSS 变量、颜色值。绝不改色
- **布局网格** — grid 列数、flex 结构、绝对定位。不重构
- **CSS 类** — `.cover`, `.content`, `.quote` 等。承载视觉标识
- **装饰元素** — 笔触纹理、水墨晕染、金色边框、装饰曲线、印章等
- **导航** — 键盘控制、底部导航条

### 始终替换（用户的内容）

- **标题** — `<h1>`, `<h2>`, `<h3>` 等
- **正文** — `<p>`, 列表项、说明文字
- **名称、日期** — 作者、日期、引文来源
- **占位内容** — 将所有模板样例内容替换为用户的实际内容
- **图片占位符** — 用 `https://trae-api-cn.mchost.guru/api/ide/v1/text_to_image?prompt={prompt}&image_size=square_hd` 或其他方式生成真实图片

### 增加/删除页面

- **增加**：从现有页面中复制最合适的布局，替换内容，更新页码
- **删除**：从末尾删除不需要的页面，更新页码

### 设计缺失的布局

如果用户需要模板中没有的布局（例如需要一个对比表格但模板只有流程页），**用模板的现有设计系统从零设计**：

- **相同字体** — 使用模板的 font-family 声明
- **相同配色** — 使用 `:root` CSS 变量
- **相同装饰语言** — 如果模板有角标、纹理、SVG 等，新页面也使用
- **相同间距** — 使用模板的 padding、margin 数值
- **相同导航** — 新页面要融入模板的导航系统

好的测试：将新页面放在两个现有页面之间打开，看起来像原装的就对了。

## 4. 常见问题

- **不要跳过询问**。即使需求很明确，用户的偏好往往超乎预期
- **不要跳过预览**。首屏是让用户感受风格最快的方式
- **不要替换字体**。"Inter 差不多够了"——不行，字体就是设计系统
- **不要改色**。即使小小的色相偏移也会破坏整体协调
- **不要混合不同模板的布局**
- **不要移除装饰元素**。它们就是风格的一部分
- **不要忘记在浏览器中打开文件**

## 5. 输出契约

每完成一个成果物（预览、迭代、最终版）：
1. `open /absolute/path/to/file.html`
2. 将绝对路径发送给用户

对于最终版，还需要在消息中包含：
- 一句话说明选了哪个模板及理由（风格匹配）
- 任何注意事项（如"我为你新设计了第 4 页的对比表格"）
